# v2.0 Scope — Trailmark Knowledge Board

Scoping for the v2.0 roadmap items (PRD §5, TASKS.md "Post-MVP"): **shareable
read-only board links**, **move cards between boards**, **tagging / filtering**,
and **card-list virtualization**. Grounded in the shipped MVP + v1.1 codebase
(all live at `knowledge-board-lovat.vercel.app`, Supabase project
`bruzjjsqcmmzptamkhrw`).

**Legend:** effort S = ≤½ day · M = ~1–2 days · L = ~3+ days (incl. migration +
security pass).

**Key difference from v1.1:** v1.1 was three render-layer features with **no DB
migrations**. v2.0 is the opposite — three of the four touch the schema, and
shareable links add the app's **first public (unauthenticated) read path**, so
security is the dominant concern, not a footnote.

**Recommended build order:** Move cards → Tagging/filtering → Shareable links →
Virtualization (cheapest/lowest-risk first; shareable links last of the built
set because it needs a security-definer read path + its own security pass;
virtualization only if usage data justifies it).

## Feature 1 — Move cards between boards  ·  Effort: M

**Why it's first:** no new public surface, no schema migration — just a scoped
`UPDATE`. A card already belongs to one board via `cards.board_id`; "move" =
repoint `board_id` and give it a fresh `order_index` at the destination.

**Approach**
- New server action `moveCard(cardId, destBoardId)` in `card-actions.ts`:
  validate the user owns **both** the source and destination board, then set
  `board_id = destBoardId` and `order_index = max(dest.order_index) + 1`
  (append to the destination end — reuse the existing `createCard` max+1 logic).
- **RLS already covers this** — the "own cards" policy's `with check` verifies
  the *new* `board_id` belongs to the user, so a move into a board you don't own
  is rejected at the DB. No policy change.
- **Attachments follow for free:** they reference `card_id` (unchanged by a
  move), and the Storage object path is `{card_id}/…` — also unchanged. No
  storage move, no re-signing.
- UI: a "Move to board…" selector in `CardDetailModal` (or `CardModal` edit) — a
  dropdown of the user's other boards. On confirm, optimistic remove-from-source
  / add-to-destination in `KnowledgeBoardApp`'s card state, then revalidate.

**Files:** `card-actions.ts` (new action), `CardDetailModal.tsx` /
`CardModal.tsx` (selector), `KnowledgeBoardApp.tsx` (optimistic state move),
`globals.css`. **No migration.**

**Risks / notes**
- Optimistic move across two boards' state arrays is the fiddly bit — on failure,
  snap both boards back (the same pattern the reorder banner already uses).
- Drag-across-boards (vs. a dropdown) is a much bigger dnd-kit lift (cross-list
  sortable) — **out of scope**; a dropdown move is the v2.0 target.

**Acceptance**
- [ ] Moving a card relocates it to the destination board, appended at the end
- [ ] Its attachments and URL survive the move (no re-upload, links still open)
- [ ] Moving into a board you don't own is rejected server-side (RLS)
- [ ] Optimistic UI; failure snaps both boards back to server truth
- [ ] `tsc` / `eslint --max-warnings=0` / `next build` clean

## Feature 2 — Tagging & filtering  ·  Effort: M–L

**Current:** cards have `title`, `description`, `url`, `status`, `order_index`.
No tags. Filtering today is only the v1.1 client-side search.

**Schema decision:** a **`tags text[]`** array column on `cards` (with a GIN
index), **not** a normalized `tags` + `card_tags` join. For a single-user
personal app the array is far simpler (no join, no extra RLS surface, trivial
client filtering), and Postgres arrays + GIN handle "cards having tag X"
efficiently. Revisit a join table only if tags need their own metadata
(color, rename-propagation) — a v2.1 concern.

**Migration** (the first schema change since MVP):
```sql
alter table public.cards
  add column tags text[] not null default '{}';
create index if not exists cards_tags_gin on public.cards using gin (tags);
```
Apply live via a named migration **and** mirror in `supabase/schema.sql`
(the project's standing rule), then regenerate `database.types.ts`.

**Approach**
- Editing: a chip/tag input in `CardModal.tsx` (add on Enter/comma, remove on
  ×). Normalize server-side in `createCard`/`updateCard`: trim, lowercase,
  de-dupe, cap length (~30 chars) and count (~10 tags/card) to bound abuse.
- Display: tag chips on the card detail (and optionally a compact row on
  `StepCard`).
- Filtering: **client-side** over `initialBoards` (same rationale as v1.1
  search — the data's already in memory). A tag filter bar (active board or
  global) narrows the visible cards; can share UI with the search input.

**Files:** migration + `schema.sql` + `database.types.ts`, `card-actions.ts`
(normalize/validate tags), `CardModal.tsx` (input), `CardDetailModal.tsx` /
`StepCard.tsx` (display), `KnowledgeBoardApp.tsx` + maybe `CardSearch.tsx`
(filter state), `types.ts` (`tags: string[]` on the card type), `globals.css`.

**Risks / notes**
- Tags are user strings rendered into the DOM — rely on React's default text
  escaping (no `dangerouslySetInnerHTML`); enforce the length/count caps
  server-side so a malicious client can't store unbounded arrays.
- Filtering + v1.1 search should compose (tag filter AND text query), not fight.

**Acceptance**
- [ ] Add/remove tags on a card; they persist across reload
- [ ] Server-side normalization + caps (length, count) enforced regardless of client
- [ ] Filtering by a tag narrows the visible cards; composes with search
- [ ] Migration applied live **and** mirrored in `schema.sql`; types regenerated
- [ ] `tsc` / `eslint --max-warnings=0` / `next build` clean

## Feature 3 — Shareable read-only board links  ·  Effort: L  ·  🔒 security-critical

**This is the headline feature and the risky one.** Every RLS policy today is
strictly owner-only (`auth.uid() = user_id`). A share link means an
**unauthenticated** visitor must read one specific board + its cards (+ maybe
attachments) — the app's first public read path. Get this wrong and it's a data
leak, so it gets its own security pass (project rule: security is top priority).

**Design — token, not "public" flag**
- Add a nullable, unique **`share_token uuid`** to `boards`. `null` = private
  (default). Generating a share sets a random v4 uuid (122 bits, unguessable);
  **revoking = null it**; **rotating = regenerate** (old link dies instantly).
  No enumeration surface — there's no "list public boards".

**Design — the read path (the crux)**
Do **not** loosen table RLS to allow anonymous SELECT directly — that's too
blunt and easy to get wrong. Instead expose exactly one controlled path:
- A **`security definer`** Postgres function `get_shared_board(token uuid)` that
  returns the board + its cards **only if** a board with that `share_token`
  exists, and is granted to the `anon` role. It's the *single* public read path;
  direct table access stays owner-only. Pin its `search_path = ''` (same
  advisor rule already applied to `touch_updated_at`).
- A public route **`/share/[token]`** (added to `PUBLIC_PATHS` in `proxy.ts`)
  that calls the function and renders a **read-only** timeline — reuse the
  presentational card/timeline components with **all edit/drag/status/delete
  affordances stripped** (no server actions wired in). Read-only is enforced
  server-side by the function only ever `SELECT`ing — never trust the UI alone.

**Attachments in shares — explicit sub-decision (recommend: defer)**
Storage RLS is owner-only and objects are private. Serving attachment previews
on a public page means minting signed URLs for a non-owner, which needs a
token-gated server route (and careful scoping). **Recommendation:** v2.0 shares
expose **text content only** (title, description-markdown, url, status, tags);
attachments are **owner-only**. Public attachments become a follow-up once the
text-share path is proven. (Call this out in the UI: "attachments are private".)

**Markdown on a public page** — v1.1 already bakes in the safe render (no
`rehype-raw`, `javascript:` stripped, links `rel="noreferrer noopener"`); that
was designed *for exactly this moment*, so shared descriptions are already safe.

**Migration**
```sql
alter table public.boards add column share_token uuid unique;
-- + create function get_shared_board(uuid) ... security definer, search_path=''
-- + grant execute to anon
```
Live migration + mirror in `schema.sql` + regenerate types.

**Files:** migration + `schema.sql` + `database.types.ts`, new
`board-actions.ts` actions (`createShareLink` / `revokeShareLink`), a "Share"
control in `BoardModal`/board UI, new `src/app/share/[token]/page.tsx` +
read-only presentational variants, `proxy.ts` (`PUBLIC_PATHS`), `globals.css`.

**Risks / notes**
- **The whole feature is a trust-boundary change** — treat the definer function
  as the security perimeter; review it like the storage policies were reviewed.
- Token in the URL leaks via referer/history/sharing — that's the accepted model
  for "anyone with the link"; rotation is the mitigation. Document it.
- No write path, ever, on the public route — no forms, no actions imported.
- Consider a lightweight rate-limit on `/share/[token]` (bonus, not blocking).
- SEO: shared pages should be `noindex` unless the user opts in (default private).

**Acceptance**
- [ ] Owner can generate, copy, rotate, and revoke a board's share link
- [ ] `/share/[token]` renders the board read-only for a signed-**out** visitor
- [ ] A wrong/revoked/rotated token returns not-found (no data, no enumeration)
- [ ] No edit/drag/status/delete affordance or server action on the public page
- [ ] Attachments are **not** exposed (text/markdown/url/tags only) in v2.0
- [ ] Markdown on the public page is XSS-safe (re-verify with the v1.1 tests)
- [ ] Focused security review of the definer function + public route
- [ ] Migration applied live **and** mirrored in `schema.sql`; types regenerated

## Feature 4 — Card-list virtualization  ·  Effort: M  ·  conditional

**Build only if usage data justifies it.** PRD §5 gates this explicitly: build
"if usage data shows boards regularly exceeding ~100 cards." The MVP flat render
is fine at the designed 10s–100 cards/board scale. **Recommendation: do NOT
schedule this yet** — keep it a documented build-if-needed item.

**Why it's non-trivial:** virtualized (windowed) lists + `@dnd-kit` sortable is
a known hard combination — the sortable context assumes all items are mounted,
so windowing breaks drag math unless carefully integrated (`@tanstack/virtual`
+ custom sortable measuring, or a dnd-kit windowing recipe). Not worth the
complexity or regression risk without evidence of the problem.

**Trigger to revisit:** real boards regularly past ~100–150 cards, or a measured
scroll/drag jank report. Until then, this stays parked.

**Acceptance (if/when built)**
- [ ] Boards with 500+ cards scroll and drag-reorder smoothly
- [ ] Reorder persistence + keyboard drag still work under virtualization
- [ ] No regression for small boards (windowing off or transparent below N)

## Cross-cutting

- **Migrations required** (unlike v1.1): tags (`text[]` + GIN) and shares
  (`share_token` + definer function). Move-cards and virtualization need none.
  Every migration: apply live via a **named** migration, **mirror in
  `supabase/schema.sql`**, then **regenerate `database.types.ts`** (standing
  project rules).
- **Security is the theme, not a footnote.** Shareable links are a
  trust-boundary change (first public read path) — they warrant a dedicated
  security review of the `security definer` function and the public route, the
  way the storage policies were scrutinized in Phase 4. Tagging needs
  server-side normalization/caps. Move-cards leans on existing RLS (verify the
  `with check` on the destination board).
- **Reuse over rebuild:** move-cards reuses the reorder optimistic pattern;
  tag-filtering reuses the v1.1 client-side search infra; the public share page
  reuses the v1.1 safe-markdown render and the presentational card components.
- **Suggested sequencing:** ship **Move cards** and **Tagging** as small,
  self-contained PRs (user-visible, low risk), then take on **Shareable links**
  as its own PR with a security pass, and treat **Virtualization** as parked
  until usage data asks for it.
- **Already shipped, not v2.0:** PRD §5 lists "lightweight progress marking
  (done state per card)" under v2.0 — that already landed in the MVP (the
  timeline status cycle next up → in progress → done). No work remains there.
- **Out of scope for v2.0** (later / only-if-asked): full-text server-side
  search, a normalized tag system with metadata, public attachments on shares,
  drag-across-boards, real-time multi-device sync, any gamification layer.
