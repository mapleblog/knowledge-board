## Feature 1 ‚Äî Search across a user's cards  ¬∑  Effort: S

**Why it's cheap:** `src/app/page.tsx` already fetches *every* board with its
cards and attachments and hands them to the `KnowledgeBoardApp` client
component (`initialBoards`). Search can be **100% client-side** over data that's
already in memory ‚Äî no new query, no server action, no DB/RLS change.

**Approach**
- Add a search input to the top bar (`KnowledgeBoardApp.tsx`, next to the brand
  / "+ New board"). On ‚â§520px it lives in the existing hamburger dropdown.
- Filter cards by case-insensitive substring match on `title`, `description`,
  and `url` across all `initialBoards`.
- Results UI: a dropdown/overlay list grouped by board; each row shows board
  accent + card title + a matched-text snippet. Clicking a result selects that
  board and opens the card's `CardDetailModal` (both already exist).
- Empty state ("No cards match '<query>'") and a clear/Escape-to-close.
- Keyboard: `/` or `Ctrl/Cmd+K` to focus; arrow keys + Enter to navigate results.

**Files:** `KnowledgeBoardApp.tsx` (state + input + results), a new
`CardSearch.tsx` component, `globals.css` (results styling). No backend files.

**Risks / notes**
- Scales fine for the expected data size; PRD already earmarks **list
  virtualization for v2** if a user exceeds ~100 cards. If search ever needs to
  cover data not loaded client-side, upgrade path is a Postgres full-text
  (`tsvector`) index + a server action ‚Äî out of scope for v1.1.
- Snippet highlighting is the only fiddly bit (escape the query for the
  highlight regex).

**Acceptance**
- [ ] Typing filters across all boards' cards by title/description/url
- [ ] Clicking a result opens that card's detail modal on the right board
- [ ] `/` or `Cmd+K` focuses search; Escape closes; empty state shows
- [ ] Works on a phone-sized viewport (input in the hamburger menu)

## Feature 2 ‚Äî Markdown in card descriptions  ¬∑  Effort: S‚ÄìM

**Current:** descriptions are stored as plain text and rendered literally ‚Äî
`CardDetailModal.tsx` does `{card.description && <p className="tag">{card.description}</p>}`.
Editing is a plain `<textarea>` in `CardModal.tsx`. **Storage stays unchanged**
(the markdown *source* is the plain-text column) ‚Äî this is a render-layer change,
no DB migration.

**Approach**
- Add `react-markdown` + `remark-gfm` (GitHub-flavored: tables, lists,
  strikethrough, task lists).
- New `Markdown.tsx` wrapper that renders `card.description` in the detail
  modal. **Do NOT enable `rehype-raw`** ‚Äî react-markdown escapes raw HTML by
  default, which is what keeps stored markdown from becoming stored XSS.
- Style the rendered elements (headings, lists, `code`, `blockquote`, links) in
  `globals.css` scoped to a `.markdown` container.
- Timeline card preview (`StepCard.tsx`) should stay **plain text** ‚Äî show a
  truncated raw snippet, not rendered markdown, so cards stay uniform.
- Editing UX (minimal): keep the textarea, add a "Markdown supported" hint.
  Optional stretch: a write/preview tab toggle in `CardModal`.

**Security (critical ‚Äî this is a stored-content render path)**
- react-markdown escapes HTML and, via its default `urlTransform`, strips
  dangerous link protocols (`javascript:` etc.) ‚Äî keep both defaults on.
- Links rendered from markdown must carry `target="_blank"
  rel="noreferrer noopener"` (match the existing `card.url` anchor).
- This matters more once **v2 shareable board links** land (descriptions become
  viewable by non-owners) ‚Äî bake the safe defaults in now. Note: server-side
  URL validation already guards the separate `card.url` field
  (`isSafeHttpUrl` in `card-actions.ts`); markdown links are a *new* surface, so
  they rely on react-markdown's sanitizer.

**Files:** `package.json` (deps), new `Markdown.tsx`, `CardDetailModal.tsx`,
optional `CardModal.tsx` (preview toggle), `globals.css`.

**Risks / notes**
- Bundle size (react-markdown + remark-gfm ‚âà modest; acceptable, code-split with
  the modal if needed).
- CSS surface: rendered markdown needs deliberate styling or it looks unstyled.

**Acceptance**
- [ ] Description markdown (headings, lists, links, code, tables) renders in the
      card detail modal
- [ ] Raw HTML in a description is escaped, not executed (XSS test:
      `<img src=x onerror=alert(1)>` and `[x](javascript:alert(1))` are inert)
- [ ] Markdown links open in a new tab with `rel="noreferrer noopener"`
- [ ] Timeline card preview stays plain text
- [ ] `tsc` / `eslint --max-warnings=0` / `next build` clean

## Feature 3 ‚Äî Google OAuth login  ¬∑  Effort: M‚ÄìL (mostly external config)

**Current:** email/password only, via server actions in `auth-actions.ts`
(`signInWithPassword`, `signUp`). Email confirmation is handled by the
`/auth/confirm` route using `verifyOtp`. OAuth uses a **different** exchange
(`exchangeCodeForSession`), so it needs its own callback route.

**Approach ‚Äî code**
- New route `src/app/auth/callback/route.ts`: reads `?code`, calls
  `supabase.auth.exchangeCodeForSession(code)`, redirects to `/` on success or
  `/login?error=oauth-failed` on error. Mirror the origin-based redirect style
  already in `auth/confirm/route.ts` so it works on any domain.
- Kick off the flow with `supabase.auth.signInWithOAuth({ provider: 'google',
  options: { redirectTo: `${origin}/auth/callback` } })`. This returns a URL to
  redirect the browser to ‚Äî trigger it from a small **client** handler (the
  browser client in `src/lib/supabase/client.ts`), since it needs
  `window.location`.
- "Continue with Google" button on both `LoginForm.tsx` and `signup/page.tsx`,
  with an "or" divider above the existing email/password form.
- RLS/data model: **no change** ‚Äî OAuth users get the same `auth.uid()`, so
  boards/cards/attachments policies already apply.

**Approach ‚Äî external setup (the long pole; account-gated, like the deploy)**
1. Google Cloud Console: create a project ‚Üí configure the OAuth **consent
   screen** ‚Üí create an **OAuth 2.0 Client ID** (Web). Authorized redirect URI
   is Supabase's callback: `https://<project-ref>.supabase.co/auth/v1/callback`.
2. Supabase dashboard ‚Üí Authentication ‚Üí Providers ‚Üí **Google**: paste the
   client ID + secret, enable.
3. Confirm the app's own redirect URLs (already set for the deploy) include the
   prod domain + localhost + preview wildcard.

**Files:** new `auth/callback/route.ts`, `LoginForm.tsx`, `signup/page.tsx`,
possibly a small `GoogleButton.tsx`, `globals.css` (button + divider). Plus the
external Google/Supabase config (no code).

**Risks / notes**
- **Redirect URI mismatch** is the classic failure ‚Äî must be exact across
  localhost / preview / prod. Test all three.
- **Account linking:** a user who signed up with email/password and later uses
  Google on the same address ‚Äî Supabase's identity-linking behavior depends on
  dashboard settings; decide and document the expected behavior (default is
  usually to link by verified email, but confirm).
- Anti-enumeration concerns (the reason the email flows return generic errors)
  don't apply to the OAuth button itself.
- Consent-screen verification: an "unverified app" warning is fine for personal
  use / limited users; publishing/verification only matters at scale.

**Acceptance**
- [ ] "Continue with Google" on login + signup completes the round-trip and
      lands an authenticated session on `/`
- [ ] Works on localhost, a Vercel preview, and production
- [ ] A Google user sees only their own boards (RLS unchanged ‚Äî spot-check)
- [ ] OAuth failure redirects to `/login` with a generic error
- [ ] Decide + document email/Google account-linking behavior

## Cross-cutting

- **No DB migrations** for any of the three (search + markdown are render-layer;
  OAuth reuses the existing identity/RLS model).
- **Security-sensitive items:** markdown render (XSS) and OAuth redirect/linking
  ‚Äî both warrant a focused pass (project rule: security is top priority).
- **Suggested sequencing:** ship Search and Markdown as one small PR each
  (fast, self-contained, user-visible), then tackle Google OAuth once its Google
  Cloud + Supabase config is ready ‚Äî same account-gated pattern as the Vercel
  deploy.
- **Out of scope for v1.1** (stays v2 per PRD): shareable read-only board links,
  moving cards between boards, tagging/filtering, list virtualization,
  server-side full-text search.
